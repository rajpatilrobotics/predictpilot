## Problem and Opportunity

DeepBookV3 is intentionally protocol-first. The official docs state that DeepBookV3 is a decentralized CLOB and **does not include an end-user interface**; instead, it provides trading functionality and SDKs that other apps can build on. DeepBook Predict follows the same pattern: the docs recommend that applications combine the public Predict server for rendering, Sui event or checkpoint streams for low-latency updates, and direct onchain object reads for confirmation-critical wallet flows. That gap matters because DeepBook Predict is more complex than a normal yes/no market app. Its architecture includes `OracleSVI` objects with live and settled lifecycle states, a shared `Vault` that takes the opposite side of trades, utilization and ask-bound controls, per-user `PredictManager` accounts, binary positions, and bounded vertical ranges. LP withdrawals are constrained by current vault value and payout coverage, and tradeability depends on oracle lifecycle. ## Why Sui and Why DeepBook Predict

Sui is a particularly strong fit for this product because its model is built around **objects** and **programmable transaction blocks**. The docs describe objects as the building blocks of all onchain state, with explicit ownership modes including address-owned, shared, and immutable objects. PTBs let applications compose multiple commands and Move calls into a single transaction without publishing a new package, which is the right mental model for sophisticated DeFi flows that need clearer execution, coin handling, and wallet review. SuiŌĆÖs examples also emphasize multi-step PTB composition as a core app pattern. ŅłĆciteŅłéturn12view3Ņłéturn12view4Ņłéturn12view5Ņłéturn12view6Ņłéturn12view7Ņłü

That architectural model maps well to DeepBook Predict specifically. Predict revolves around shared protocol state (`Predict`, `Vault`, `OracleSVI`) plus a reusable per-user `PredictManager`. The protocol documentation spells out the core flows that apps need to orchestrate: create a manager, preview binary or range amounts, mint or redeem positions, and supply or withdraw vault liquidity. For page rendering, the official public server already exposes state, oracle, quote-asset, vault summary, manager summary, position summary, PnL, and history endpoints. This combination of object-native execution and indexed data is exactly what a ŌĆ£terminalŌĆØ product wants. ŅłĆciteŅłéturn1view1Ņłéturn13view0Ņłéturn13view2Ņłéturn11search2Ņłü

Sui also lowers the UX ceiling for a polished app. The official Sui dApp Kit automatically discovers Wallet Standard wallets, provides React and framework-agnostic packages, and supports connection and transaction execution flows. Beyond that, Sui offers native **zkLogin** for OAuth-style onboarding and **sponsored transactions** for gas abstraction, both of which exist to reduce friction for new users. PredictPilot does not need to depend on those in the MVP, but they form a credible path from hackathon demo to broad adoption. ŅłĆciteŅłéturn12view0Ņłéturn12view1Ņłéturn23view1Ņłéturn23view0Ņłü

## Target Users, Pain Points, and Product Thesis

PredictPilot is for three primary user groups. First are **traders** who want to explore binary and range positions without reading raw contract docs. Second are **liquidity providers** who need to understand vault value, `PLP`, withdrawal conditions, and exposure before contributing funds. Third are **judges, developers, and ecosystem operators** who need to see DeepBook PredictŌĆÖs primitives demonstrated clearly, not buried behind unlabeled charts or arbitrary buttons. The official docs already distinguish trader flows, liquidity-provider flows, and indexed portfolio/vault/history surfaces, which makes these user groups a natural fit. The core pain points are straightforward. Users do not naturally understand `OracleSVI`, vertical ranges, settlement lifecycle, vault max-payout coverage, or per-user shared managers. Raw onchain reads are too low-level for everyday product UX, but purely indexed views are not sufficient when a wallet flow needs authoritative confirmation. ## Success Metrics, UX Principles, and Trust Model

The **north star metric** for PredictPilot is: **successful decision-to-execution sessions**. In practical terms, that means a user can understand a market, create or reuse a `PredictManager`, fund it with the accepted quote asset, preview a trade or LP action, sign the transaction, and see the resulting state reflected correctly afterward.

The **demo success criteria** are intentionally concrete. A winning demo should show: a live market and oracle page; managed portfolio and vault summaries from the official Predict server; at least one successful binary-position mint and redeem flow; at least one successful vertical-range flow; and at least one LP supply or withdrawal flow featuring `PLP` and vault-aware constraints. Those flows mirror the public functions and server endpoints documented today. ŅłĆciteŅłéturn11search2Ņłéturn13view0Ņłéturn13view2Ņłü

The **technical success criteria** are equally important. PredictPilot should load package IDs, object IDs, quote assets, and server URLs from environment-driven config rather than fossilizing them in code, because the official docs explicitly say the current testnet integration surface is provisional. It should use the Predict server for render-ready data, rely on direct onchain reads around confirmation-critical flows, and expose network/version context visibly in the UI. PTB construction should be explicit and inspectable, not hidden behind magical abstractions. ŅłĆciteŅłéturn1view0Ņłéturn1view2Ņłéturn12view5Ņłéturn12view6Ņłü

The **user-experience principles** are: explain before asking for action; show protocol state before wallet signature; make the consequences of settlement, range bounds, LP coverage, and vault constraints visible; and keep object complexity behind good interface design without pretending it does not exist. Wallet flows should be standards-based through Sui dApp Kit and the Wallet Standard, and any future onboarding shortcuts should build on official mechanisms such as zkLogin or sponsored transactions rather than custom custodial shortcuts.
